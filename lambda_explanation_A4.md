# Как подбирается lambda простыми словами

Lambda нужна, чтобы программа не исправляла измерения слишком слабо и не исправляла их слишком сильно.

Есть две задачи одновременно.

Первая задача: сделать так, чтобы баланс сходился.

То есть сколько вещества вошло в узел, примерно столько же должно выйти.

Если баланс не сходится, значит в измерениях есть ошибка или шум.

Вторая задача: не испортить реальные измерения.

Датчики уже что-то измерили, поэтому нельзя просто взять и сильно поменять все потоки.

Нужно менять только настолько, насколько это действительно нужно для сведения баланса.

Lambda как раз управляет этим компромиссом.

Если lambda слишком маленькая, программа слишком свободно меняет измеренные потоки.

Баланс может хорошо сойтись, но значения станут слишком далекими от показаний датчиков.

Это плохо, потому что мы начинаем слишком сильно переделывать исходные данные.

Если lambda слишком большая, программа почти не трогает измерения.

Тогда значения остаются близкими к датчикам, но баланс может сходиться плохо.

Это тоже плохо, потому что задача сведения баланса не решается.

Поэтому нужно найти не минимальную и не максимальную lambda, а нормальную середину.

## Формула из книги

Из книги взята не готовая формула для вычисления lambda, а сама постановка задачи регуляризации.

В книге смысл такой:

```text
F = ошибка баланса + lambda * поправка измерений -> минимум
```

Если записать ближе к книге:

```text
F(V, lambda) = |A * V|^2 + lambda * |V - V0|^2 -> min
```

Где:

```text
V  - исправленные потоки;
V0 - исходные измеренные потоки;
A * V - остаток материального баланса;
lambda - насколько сильно программа должна держаться за исходные измерения.
```

То есть из книги взята главная идея:

```text
исправить баланс, но не уходить слишком далеко от измерений.
```

## Формула подбора lambda в проекте

Сначала для каждой lambda решается задача Тихонова:

```text
x(lambda) = argmin [ B(x) + lambda * D(x) ]

B(x) = ||(A * x - b) / sigma_узл||^2

D(x) = ||(x - x0) / sigma||^2
```

Потом строится L-кривая:

```text
X(lambda) = lg sqrt(B(x(lambda)))

Y(lambda) = lg sqrt(D(x(lambda)))
```

И lambda выбирается по углу L-кривой:

```text
lambda* = argmax curvature( X(lambda), Y(lambda) )
```

По-человечески:

```text
берем ту lambda, где L-кривая сильнее всего сгибается.
```

Это взято из метода L-кривой: L-кривая строится как зависимость логарифма нормы невязки от логарифма нормы стабилизатора.

Обозначения:

```text
x(lambda) - исправленные потоки при конкретной lambda;
x0 - исходные измерения;
B(x) - остаток баланса;
D(x) - насколько сильно изменили измерения;
sigma - погрешность датчика;
curvature - кривизна линии;
lambda* - выбранная lambda.
```

В проекте lambda не выбирается руками.

Программа сама проверяет разные значения lambda.

Для каждого значения lambda она заново решает задачу сведения потоков.

То есть программа пробует один вариант lambda, считает исправленные потоки и смотрит результат.

Потом пробует другое значение lambda и снова смотрит результат.

Для каждого варианта программа считает две точки для L-кривой.

Первая координата: логарифм остатка баланса.

Вторая координата: логарифм величины поправок.

Дальше программа ищет угол этой кривой.

В углу получается нормальный компромисс:

баланс уже стал лучше, но измерения еще не испорчены лишними поправками.

В книге используется идея регуляризации.

Смысл такой: нельзя просто заставить баланс сойтись любой ценой.

Нужно учитывать, что измерения имеют свою точность и им тоже надо доверять.

Поэтому программа ищет такой вариант, где баланс исправлен, но измерения не изменены без необходимости.

Еще важно, что в проекте учитывается точность датчиков.

Если у датчика маленькая погрешность, то его значение нельзя сильно двигать.

Если у датчика большая погрешность, то его значение можно двигать свободнее.

Поэтому одна и та же поправка может быть нормальной для неточного датчика и подозрительной для точного датчика.

Именно поэтому в таблицах есть величина delta/sigma.

Она показывает, во сколько раз поправка больше обычной погрешности датчика.

Если значение большое, значит поток пришлось заметно исправить.

Такие потоки и считаются подозрительными.

Итог простой.

Lambda - это ручка баланса между двумя желаниями.

Первое желание: чтобы материальный баланс сошелся.

Второе желание: чтобы измерения не поменялись слишком сильно.

Программа сама перебирает варианты lambda, каждый раз сводит баланс и выбирает тот вариант, где общий результат лучший.

Поэтому lambda в проекте не взята случайно и не поставлена вручную.

Она найдена автоматически по расчету.
